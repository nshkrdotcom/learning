# 141. `engine.py`: the scalar autograd engine

The core file is:

```text id="dr2sde"
engine.py
```

This contains the `Value` class.

This is the entire scalar autograd engine.

Conceptually, `Value` contains:

```text id="w2xa8t"
data
grad
_prev
_op
_backward
```

Meaning:

```text id="nnf9t4"
data      = scalar forward value
grad      = derivative of final output with respect to this value
_prev     = previous Value objects that produced this one
_op       = operation string used for graph visualization/debugging
_backward = local function that propagates gradients to children
```

This is the complete core abstraction.

---

# 142. `Value.__init__`

The constructor stores the scalar and graph metadata:

```python id="r9x0bl"
class Value:
    def __init__(self, data, _children=(), _op='', label=''):
        self.data = data
        self.grad = 0.0
        self._backward = lambda: None
        self._prev = set(_children)
        self._op = _op
        self.label = label
```

The important decisions:

```text id="bo83ob"
grad starts at 0
_backward defaults to no-op
_prev records children
_op records the operation that created the node
```

Leaf nodes have:

```text id="82y740"
_children = ()
_op = ''
_backward = no-op
```

Operation outputs have:

```text id="5ld056"
_children = inputs to the operation
_op = operation name
_backward = local chain-rule function
```

# 144. Addition in `engine.py`

Addition:

```python id="nlr0jy"
def __add__(self, other):
    other = other if isinstance(other, Value) else Value(other)
    out = Value(self.data + other.data, (self, other), '+')

    def _backward():
        self.grad += out.grad
        other.grad += out.grad

    out._backward = _backward
    return out
```

Forward:

```text id="ib5k3e"
out.data = self.data + other.data
```

Graph:

```text id="363d7t"
out._prev = {self, other}
out._op = '+'
```

Backward:

```text id="byhegh"
∂out/∂self = 1
∂out/∂other = 1
```

So:

```text id="bjr291"
self.grad += out.grad
other.grad += out.grad
```

Key detail:

```text id="gsrmn2"
+=, not =
```

because a node may receive gradient contributions through multiple paths.

# 145. Multiplication in `engine.py`

Multiplication:

```python id="co9n00"
def __mul__(self, other):
    other = other if isinstance(other, Value) else Value(other)
    out = Value(self.data * other.data, (self, other), '*')

    def _backward():
        self.grad += other.data * out.grad
        other.grad += self.data * out.grad

    out._backward = _backward
    return out
```

Forward:

```text id="j2s8sl"
out.data = self.data * other.data
```

Local derivatives:

```text id="4b6um6"
∂out/∂self = other.data
∂out/∂other = self.data
```

Backward:

```text id="pp8sw8"
self.grad += other.data * out.grad
other.grad += self.data * out.grad
```

This is multiplication’s local chain rule.

# 146. Power in `engine.py`

Power by a scalar constant:

```python id="wk4sol"
def __pow__(self, other):
    assert isinstance(other, (int, float))
    out = Value(self.data**other, (self,), f'**{other}')

    def _backward():
        self.grad += other * (self.data ** (other - 1)) * out.grad

    out._backward = _backward
    return out
```

This supports expressions like:

```python id="35o536"
x ** 2
x ** -1
```

Derivative rule:

```text id="u0r925"
d/dx xⁿ = n*xⁿ⁻¹
```

So the backward formula is:

```text id="rjf1uw"
incoming gradient * local derivative
= out.grad * other * self.data**(other-1)
```

The exponent is required to be a plain Python number, not another `Value`.

That keeps the derivative rule simple.

# 147. ReLU in actual micrograd

In the notebook, Karpathy used `tanh`.

In the actual repository, the nonlinearity shown is `relu`.

ReLU means:

```text id="1x4gjv"
relu(x) = max(0, x)
```

Forward:

```text id="xd35af"
if x < 0: relu(x) = 0
if x > 0: relu(x) = x
```

Backward:

```text id="xi551b"
if output > 0: gradient passes through
if output <= 0: gradient becomes 0
```

Conceptual implementation:

```python id="g4beau"
def relu(self):
    out = Value(0 if self.data < 0 else self.data, (self,), 'ReLU')

    def _backward():
        self.grad += (out.data > 0) * out.grad

    out._backward = _backward
    return out
```

The exact nonlinearity is not the central issue.

The important pattern is the same:

```text id="zsr56p"
define forward value
define local derivative
chain incoming gradient backward
```

Karpathy notes that `tanh`, `relu`, sigmoid, and similar functions are all nonlinearities commonly used in MLPs.

# 148. Other convenience operations in `engine.py`

The actual `Value` class also defines arithmetic conveniences using the primitives.

Examples:

```python id="63rfrw"
def __neg__(self):
    return self * -1

def __sub__(self, other):
    return self + (-other)

def __radd__(self, other):
    return self + other

def __rmul__(self, other):
    return self * other

def __truediv__(self, other):
    return self * other**-1

def __rtruediv__(self, other):
    return other * self**-1
```

These are mostly syntactic conveniences.

They make expressions like these work:

```python id="uigovg"
a + 1
1 + a
a - b
-a
a * 2
2 * a
a / b
2 / a
```

Most are built out of already-defined differentiable operations.

This avoids duplicating backward formulas.

# 149. `Value.backward()` in `engine.py`

The public backward method performs reverse-mode autodiff.

Conceptually:

```python id="iu9yhq"
def backward(self):

    topo = []
    visited = set()

    def build_topo(v):
        if v not in visited:
            visited.add(v)
            for child in v._prev:
                build_topo(child)
            topo.append(v)

    build_topo(self)

    self.grad = 1.0
    for v in reversed(topo):
        v._backward()
```

Steps:

```text id="d5wjnb"
1. topologically sort all nodes leading into self
2. set self.grad = 1
3. traverse nodes in reverse topological order
4. call each node’s local _backward
```

This is the whole autograd mechanism.

---

# 150. Why `backward()` starts with `self.grad = 1`

If calling:

```python id="u78siz"
loss.backward()
```

then `self` is the loss.

The derivative of the loss with respect to itself is:

```text id="f15gx7"
∂loss/∂loss = 1
```

So:

```python id="eeuuqa"
self.grad = 1
```

is the seed value for the reverse pass.

Everything else is obtained by chain rule.

# 152. `Module` base class

The actual micrograd code includes a base class:

```python id="72s221"
class Module:
    def zero_grad(self):
        for p in self.parameters():
            p.grad = 0

    def parameters(self):
        return []
```

This mirrors PyTorch’s `nn.Module`.

Purpose:

```text id="4jruq2"
provide common behavior for all neural network modules
```

The most important method:

```python id="wzopqx"
zero_grad()
```

It resets all parameter gradients.

This formalizes the bug fix from the training loop.

Instead of writing:

```python id="sygzo7"
for p in n.parameters():
    p.grad = 0.0
```

you can call:

```python id="p4awdi"
n.zero_grad()
```

# 153. Why `Module.parameters()` returns `[]`

The base `Module` does not know what parameters its subclasses have.

So:

```python id="dbpl1c"
def parameters(self):
    return []
```

is a default placeholder.

Subclasses override it.

For example:

```text id="um78sf"
Neuron.parameters returns weights and bias
Layer.parameters returns parameters of all neurons
MLP.parameters returns parameters of all layers
```

# 154. `Neuron` in `nn.py`

A neuron has:

```text id="qd7y1o"
weights
bias
nonlinearity option
```

Conceptual version:

```python id="g19gk5"
class Neuron(Module):
    def __init__(self, nin, nonlin=True):
        self.w = [Value(random.uniform(-1, 1)) for _ in range(nin)]
        self.b = Value(0)
        self.nonlin = nonlin
```

Notes:

```text id="9t1dic"
nin = number of inputs
w = one scalar weight per input
b = scalar bias
nonlin = whether to apply activation function
```

In the notebook, the bias was randomly initialized.

In the repository version, Karpathy’s code may use a simple bias initialization such as zero.

The exact initialization is not central for the lecture’s conceptual flow.

# 155. `Neuron.__call__`

Forward pass:

```python id="ufk434"
def __call__(self, x):
    act = sum((wi*xi for wi, xi in zip(self.w, x)), self.b)
    return act.relu() if self.nonlin else act
```

The neuron computes:

```text id="vjt4sy"
act = w · x + b
```

Then either:

```text id="wlswlq"
return relu(act)
```

or, if it is the output layer:

```text id="kl80ay"
return act
```

The `nonlin` flag lets the final layer be linear.

This is common in neural networks:

```text id="33x46t"
hidden layers often use nonlinearities
final layer may or may not use one depending on the task
```

# 157. `Layer` in `nn.py`

A layer is a list of neurons:

```python id="df0e1a"
class Layer(Module):
    def __init__(self, nin, nout, **kwargs):
        self.neurons = [Neuron(nin, **kwargs) for _ in range(nout)]
```

Here:

```text id="6zzjgh"
nin = number of inputs to each neuron
nout = number of neurons in the layer
```

Forward pass:

```python id="gpclwi"
def __call__(self, x):
    out = [n(x) for n in self.neurons]
    return out[0] if len(out) == 1 else out
```

So a layer maps:

```text id="xlxh4c"
input vector of length nin
→ output vector of length nout
```

If `nout = 1`, return the single scalar directly.

---

# 158. `Layer.parameters()`

A layer’s parameters are all parameters of all its neurons:

```python id="tbe1j1"
def parameters(self):
    return [p for n in self.neurons for p in n.parameters()]
```

This flattens the nested structure:

```text id="wzpfu2"
layer
→ neurons
→ weights and biases
```

into one list.

# 159. `MLP` in `nn.py`

An MLP is a list of layers.

Conceptual version:

```python id="imk1c4"
class MLP(Module):
    def __init__(self, nin, nouts):
        sz = [nin] + nouts
        self.layers = [
            Layer(sz[i], sz[i+1], nonlin=i != len(nouts)-1)
            for i in range(len(nouts))
        ]
```

If:

```python id="a3u5m8"
MLP(3, [4, 4, 1])
```

then:

```text id="rm8sfg"
sz = [3, 4, 4, 1]
```

Layers:

```text id="dlsprf"
Layer(3, 4)
Layer(4, 4)
Layer(4, 1)
```

The final layer uses:

```text id="k8gir5"
nonlin=False
```

so the final output is linear.

Earlier in the notebook, the toy MLP used `tanh` more directly; the repository version uses ReLU-style hidden layers.

The conceptual design is the same.

# 160. `MLP.__call__`

Forward pass:

```python id="dclwpo"
def __call__(self, x):
    for layer in self.layers:
        x = layer(x)
    return x
```

Data flows sequentially:

```text id="p1bpz4"
x
→ layer 1
→ layer 2
→ ...
→ output layer
```

Each layer creates more `Value` operations.

The final output is a `Value` or list of `Value`s depending on the architecture.

---

# 161. `MLP.parameters()`

An MLP’s parameters are all layer parameters:

```python id="hkehbw"
def parameters(self):
    return [p for layer in self.layers for p in layer.parameters()]
```

This gives a flat list usable by training code:

```python id="hsxh8q"
for p in model.parameters():
    p.data += -lr * p.grad
```

This abstraction lets optimization code ignore the internal hierarchy.

# 169. Key distinction: conceptual completeness vs production completeness

Micrograd is conceptually complete for understanding backprop.

It is not production complete.

It lacks:

```text id="dik0ex"
tensor operations
GPU kernels
batching efficiency
advanced optimizers
mixed precision
distributed training
large operation library
memory optimization
automatic broadcasting semantics
serious numerical stability work
```

But it contains the essential conceptual ingredients:

```text id="gizw07"
differentiable values
computation graph
reverse-mode autodiff
gradient accumulation
parameterized neural nets
gradient-based optimization
```

That is why it is valuable pedagogically.
