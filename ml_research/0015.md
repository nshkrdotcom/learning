# 118. Turning manual updates into a training loop

Karpathy now makes the repeated optimization process more respectable.

Instead of manually running:

```python id="eegg7f"
ypred = [n(x) for x in xs]
loss = sum((yout - ygt)**2 for ygt, yout in zip(ys, ypred))
loss.backward()

for p in n.parameters():
    p.data += -0.01 * p.grad
```

over and over, he writes a loop:

```python id="32lser"
for k in range(20):
    # forward pass
    ypred = [n(x) for x in xs]
    loss = sum((yout - ygt)**2 for ygt, yout in zip(ys, ypred))

    # backward pass
    loss.backward()

    # update
    for p in n.parameters():
        p.data += -0.05 * p.grad

    print(k, loss.data)
```

The three core phases are:

```text id="6a5dg6"
1. forward pass
2. backward pass
3. parameter update
```

This is the basic neural-network training loop.

---

# 119. Forward pass in the loop

The forward pass computes predictions:

```python id="sih3xw"
ypred = [n(x) for x in xs]
```

This runs all four training examples through the MLP.

Then it computes the loss:

```python id="ydnjr5"
loss = sum((yout - ygt)**2 for ygt, yout in zip(ys, ypred))
```

The forward pass does two things simultaneously:

```text id="kc8ckc"
1. computes numeric prediction and loss values
2. builds the computation graph needed for backprop
```

Because all parameters are `Value` objects, every arithmetic operation records graph structure.

---

# 120. Backward pass in the loop

The backward pass:

```python id="rg163b"
loss.backward()
```

does:

```text id="jdngtg"
1. topologically sorts all nodes leading into loss
2. sets loss.grad = 1
3. walks backward through the graph
4. applies every local _backward function
5. accumulates gradients into parameters
```

After this call, every parameter has:

```text id="o9jaz0"
p.grad = ∂loss/∂p
```

These gradients describe how changing each weight or bias would locally affect the loss.

---

# 121. Parameter update in the loop

The update:

```python id="fxh1vk"
for p in n.parameters():
    p.data += -0.05 * p.grad
```

is gradient descent.

Because:

```text id="lrkszc"
gradient points uphill in loss
```

the update moves in the opposite direction:

```text id="3w56a0"
negative gradient direction
```

The learning rate controls the step size.

Karpathy uses a learning rate around:

```text id="6hr4wp"
0.05
```

after noting that:

```text id="bmizpv"
0.01 is slow
0.1 can be dangerously large
```

---

# 122. Initial apparent success

Running the loop prints losses decreasing over steps.

The predictions become close to targets:

```text id="511ztf"
target:      [ 1, -1, -1,  1]
prediction: [≈1, ≈-1, ≈-1, ≈1]
```

This looks like the network is training correctly.

But then Karpathy notices a serious bug.

The loop is missing a step.

---

# 123. The bug: forgetting to zero gradients

The bug is:

```text id="lvyx69"
gradients are not reset before each backward pass
```

This is one of the common neural-net mistakes:

```text id="320z3b"
forgot to zero_grad before backward
```

In micrograd, every parameter has:

```python id="v5m93i"
p.grad
```

When `loss.backward()` runs, local backward functions do:

```python id="v4v0st"
+=
```

not assignment.

That was necessary because gradients from multiple graph paths must accumulate during one backward pass.

But across separate training iterations, we do **not** want old gradients to remain.

---

# 124. Why gradient accumulation is needed inside one backward pass

Inside a single graph, one parameter can influence the loss through multiple paths.

Example:

```python id="t5p8h2"
a = Value(3.0)
b = a + a
```

The gradient should be:

```text id="c2pjhv"
db/da = 2
```

So when backprop reaches `a` from two branches, contributions must add.

That is why operation backward functions use:

```python id="r0enxm"
self.grad += ...
```

not:

```python id="hme3gc"
self.grad = ...
```

Within one backward pass:

```text id="2i2imn"
accumulation is correct
```

---

# 125. Why accumulation is wrong across training iterations

Training iteration 1:

```text id="woy14k"
compute loss_1
backward → p.grad contains ∂loss_1/∂p
update p.data
```

Training iteration 2:

```text id="zixqei"
compute loss_2
backward → p.grad should contain ∂loss_2/∂p only
```

But if we do not reset:

```text id="r6p904"
p.grad still contains ∂loss_1/∂p
```

Then because backward uses `+=`, after iteration 2:

```text id="fz9dzk"
p.grad = ∂loss_1/∂p + ∂loss_2/∂p
```

After many iterations:

```text id="oqvvku"
p.grad = stale accumulated gradients from all previous losses
```

That is wrong for ordinary gradient descent.

It effectively changes the update rule and can create huge unintended steps.

---

# 126. Why the buggy loop still seemed to work

Karpathy notes that the previous buggy loop still trained the tiny model.

Reason:

```text id="w7lbsl"
the problem is extremely simple
the network has enough capacity
the accumulated gradients acted like a large effective step size
```

So the model still found a good solution.

But this is dangerous.

On a harder problem, stale accumulated gradients could:

```text id="pjdrlc"
destabilize training
prevent convergence
make the loss explode
hide subtle correctness errors
```

This is common in neural-network work:

```text id="79s56s"
buggy code can appear to work on easy problems
```

---

# 127. Correct fix: zero gradients before backward

Before each backward pass, reset all parameter gradients:

```python id="11vvc6"
for p in n.parameters():
    p.grad = 0.0
```

Correct training loop:

```python id="tg69v7"
for k in range(20):
    # forward pass
    ypred = [n(x) for x in xs]
    loss = sum((yout - ygt)**2 for ygt, yout in zip(ys, ypred))

    # zero grad
    for p in n.parameters():
        p.grad = 0.0

    # backward pass
    loss.backward()

    # update
    for p in n.parameters():
        p.data += -0.05 * p.grad

    print(k, loss.data)
```

The order matters:

```text id="xnsxje"
forward
zero gradients
backward
update
```

Zeroing happens after the forward pass but before backward.

It could also happen before the forward pass, as long as it happens before `loss.backward()`.

---

# 128. Relation to PyTorch’s `zero_grad`

In PyTorch, optimizers commonly use:

```python id="2u8pnr"
optimizer.zero_grad()
loss.backward()
optimizer.step()
```

or module-level:

```python id="p1ojnu"
model.zero_grad()
```

Same reason:

```text id="g1pz5r"
PyTorch gradients accumulate by default
```

This accumulation is useful for some advanced workflows, such as accumulating gradients over multiple mini-batches.

But for standard training, you reset gradients each step.

Micrograd exposes the same issue in a small and visible way.

---

# 129. Corrected optimization is slower but cleaner

After zeroing gradients correctly, training descends more gradually.

Previously, stale gradients made the model move too aggressively.

With the corrected loop:

```text id="wvf238"
loss decreases more slowly
training behavior is more controlled
```

Karpathy increases steps or adjusts learning rate to continue reducing loss.

Eventually predictions again approach:

```text id="bajyxq"
[1, -1, -1, 1]
```

but now for the right reason.

# 130. Final form of the training loop

The canonical micrograd training loop is:

```python id="6vzj0o"
for k in range(num_steps):

    # forward pass
    ypred = [model(x) for x in xs]
    loss = sum((yout - ygt)**2 for ygt, yout in zip(ys, ypred))

    # backward pass
    for p in model.parameters():
        p.grad = 0.0
    loss.backward()

    # update
    for p in model.parameters():
        p.data += -learning_rate * p.grad
```

This is the essential deep-learning loop.

PyTorch equivalent conceptually:

```python id="6xp76m"
for batch in data:
    ypred = model(x)
    loss = loss_fn(ypred, y)

    optimizer.zero_grad()
    loss.backward()
    optimizer.step()
```

Micrograd has no optimizer object, so the update is written manually.

# 131. What neural-network training is

Karpathy summarizes neural-network training as:

```text id="dud48t"
define a mathematical expression
measure its performance using a loss
compute gradients of loss with respect to parameters
adjust parameters to reduce loss
repeat
```

The network is a parameterized function:

```text id="xkmdpt"
prediction = neural_net(data, weights)
```

The loss function measures how bad predictions are:

```text id="sr1aiu"
loss = loss_function(predictions, targets)
```

Training changes weights so that loss decreases.

---

# 132. The role of the loss function

The loss function is designed so that:

```text id="uhd7bc"
low loss = network behavior is good
high loss = network behavior is bad
```

In the toy problem:

```text id="aigbj5"
loss = sum of squared prediction errors
```

So minimizing the loss means making predictions match targets.

In larger real tasks, the loss may be different.

For language models, Karpathy mentions next-token prediction and cross-entropy loss.

But the structure remains:

```text id="ocrvn3"
model prediction
→ loss
→ backward
→ parameter update
```

---

# 133. The role of backpropagation

Backpropagation computes:

```text id="pspcxu"
∂loss/∂parameter
```

for every parameter.

This tells us:

```text id="aqwuxy"
how each parameter locally affects the loss
```

Then gradient descent uses this information:

```text id="ueckfj"
move parameters opposite the gradient
```

Backpropagation itself is:

```text id="5864yi"
recursive chain rule over a computation graph
```

Micrograd implemented this using:

```text id="91djv2"
Value objects
local _backward functions
topological sort
gradient accumulation
```

---

# 134. The role of gradient descent

Gradient descent is the iterative update procedure:

```python id="lvrwjs"
p.data += -learning_rate * p.grad
```

It uses local slope information to reduce the loss.

The gradient is local, so the process must be repeated:

```text id="tr5zyk"
small local step
recompute loss and gradients
small local step
recompute loss and gradients
...
```

This progressively improves the model.

---

# 135. The role of parameters

The weights and biases are the adjustable parts of the network.

Inputs and targets are fixed by the dataset.

Parameters are changed by training.

For the toy MLP:

```text id="sbdok4"
41 scalar parameters
```

For modern large neural networks:

```text id="b5d6g8"
billions or even hundreds of billions of parameters
```

But conceptually each parameter still has:

```text id="vi6dgn"
data
grad
```

and gets updated using gradient information.

# 137. Why micrograd is enough to understand the core

Micrograd is tiny because it strips away efficiency layers.

It shows:

```text id="0p392q"
what a value is
what a gradient is

```

Modern frameworks are larger because they must handle:


