# E002: Explainer-Prediction Baseline (pre-intervention prediction scoring)

## Status
Planned. Depends on E001 having produced real top-ranked feature IDs and
real `top_examples.jsonl` evidence to label. Can be developed in parallel
with E001's intervention code, since it consumes E001's ranking output but
doesn't depend on E001's intervention results being finished first.

## Research question
Does a structured, pre-intervention prediction about a candidate SAE
feature's causal role — generated from its top-activating examples, the
same input a standard automated-interpretability labeler would use —
predict the correct sign of the real decoded intervention's effect on the
matched target prompt, more often than chance?

## Hypothesis
Predictions derived from genuinely feature-specific top-activating examples
will predict the correct sign more often than predictions derived from a
feature with superficially similar examples but a different decoder
direction (a direct test of the "descriptive collision" concern — see
McCann 2026 in `literature/prior_art_matrix.md`). No strong prior on the
magnitude of this effect; this experiment is explicitly allowed to come
back null or weak, and a clean null result here is itself a legitimate,
citable finding about autointerp-label reliability.

## Model
The candidate SAE features come from the same pipeline as E001
(`EleutherAI/pythia-70m-deduped`, `pythia-70m-deduped-res-sm`). The
*prediction* is generated by a separate, larger instruction-following model
— Pythia-70m is too small to produce a coherent structured prediction.
Which model produced each prediction must be recorded with the prediction
itself (see provenance fields below).

## Task
For each of the top-`k` features selected in E001 (and, as a contrast
case, for one or two features McCann-style "collision" candidates — chosen
by high decoder cosine-similarity to a top feature but a different
activation profile, if any are found in the ranking artifact): take the
feature's `top_examples.jsonl` entries, generate a structured prediction,
then compare to the real measured intervention outcome from E001's runs.

## Mechanism objects
SAE features only — same `sae_N` IDs already produced by E001's ranking.

## Claim format
Each prediction is a structured, falsifiable, written-down-before-scoring
record:

```json
{
  "feature_id": "sae_12300",
  "label_source_model": "<exact model name/version used to generate this>",
  "label_generated_at": "<ISO date>",
  "label_prompt_path": "docs/explainer_prediction_prompt.md",
  "short_label": "<the model's own short description of the feature>",
  "predicted_target_direction": "increase | decrease",
  "predicted_control_direction": "increase | decrease | no_change",
  "predicted_relative_magnitude": "target > control | target <= control"
}
```

This is generated and saved *before* being compared to E001's real
intervention numbers — not reverse-engineered to match them.

## Intervention
None new — reuses the real decoded SAE intervention results already
produced by E001/Phase 2 for the same feature IDs. This experiment is a
scoring layer on top of existing real intervention output, not a new
intervention pipeline.

## Metrics
- `sign_match`: did `predicted_target_direction` match the real
  `target_signed_delta`'s sign?
- `relative_magnitude_match`: did the predicted target-vs-control ordering
  match the real `specificity_gap`'s sign?
- Aggregate sign-match rate across all labeled features, compared against
  a literal coin-flip baseline (0.5) and, if collision candidates are
  included, compared between "genuine" and "collision" feature predictions.

## Baselines
- Chance (0.5 sign-match rate).
- If available: a "collision" feature (high decoder similarity, different
  real behavior) — does its prediction get the sign wrong more often than a
  non-collision feature's prediction?

## Controls
The prediction must be written down and saved (with the provenance fields
above) before it is compared against the real outcome. No editing the
prediction after seeing the real result.

## Success criterion
Sign-match rate is reportable with the real number, whatever it is — there
is no specific number that makes this "successful." The success condition
for the *experiment* (not the hypothesis) is: predictions were generated
honestly before scoring, provenance is recorded, and the comparison ran on
real E001 intervention numbers, not placeholders.

## Failure criterion
Predictions were edited after seeing real outcomes, or provenance for the
label-generating model/prompt/date is missing, or the comparison silently
falls back to fabricated intervention numbers. Any of these invalidates the
experiment regardless of how good the resulting numbers look.

## Notes
- This is the one piece of the project that brings back the literal
  "self/explanation grounded in intervention" framing the project is named
  for — Phase 3 as originally specified only tests feature-*selection*
  validity, never a *prediction's* accuracy. See decision D011 for why this
  is implemented as an offline, recorded, versioned annotation step rather
  than a live API call inside the automated pipeline: determinism and
  reproducibility matter more here than convenience, and the existing
  Phase 3 hard rules already forbid hidden network calls in the core path.
- Check Neuronpedia first (decision D010) — if `sae_12300`/`sae_25521`
  already have a public auto-interp label, record that as an additional,
  independently-sourced prediction to compare, before generating a new one.
- Keep the labeling prompt itself in `docs/explainer_prediction_prompt.md`,
  versioned, so a reviewer can see exactly what the labeling model was
  shown.
