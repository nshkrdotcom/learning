Karpathy micrograd outline — Part 3 of 8

Scope: implementing automatic local backward functions → topological ordering → full .backward() → gradient accumulation bug → breaking up tanh into smaller ops

Approx. video range: 1:09:00–1:39:30

41. Automating the backward pass: store a local _backward function

Manual backprop worked, but it is not scalable.

Karpathy now modifies Value so every node stores a function:

self._backward = lambda: None

At construction time, a plain leaf node has no operation behind it, so its _backward function does nothing.

Example leaf nodes:

x1 = Value(2.0)
w1 = Value(-3.0)
b = Value(6.88)

These were not created by an operation inside the graph. There is no previous operation to backpropagate through.

So for a leaf:

_backward does nothing

But for operation outputs, _backward will encode the local chain-rule behavior.

42. Local backward for addition

For:

out = self + other

the forward pass creates:

out = Value(self.data + other.data, (self, other), '+')

Now we attach a backward function to out.

Addition rule:

out = self + other

∂out/∂self = 1
∂out/∂other = 1

By chain rule:

∂L/∂self  = ∂L/∂out * ∂out/∂self
           = out.grad * 1

∂L/∂other = ∂L/∂out * ∂out/∂other
           = out.grad * 1

So the local backward function is:

def _backward():
    self.grad += 1.0 * out.grad
    other.grad += 1.0 * out.grad

Then:

out._backward = _backward

Conceptually:

addition copies the incoming gradient to both inputs

43. Why _backward is a closure

The _backward function is defined inside __add__.

It closes over:

self
other
out

That means when out._backward() is called later, it still knows:

which two inputs created out
what out.grad is
how to update self.grad and other.grad

This is key.

The graph stores not just numeric values and pointers, but also the local recipe for backpropagating each operation.

44. Local backward for multiplication

For:

out = self * other

the local derivatives are:

∂out/∂self = other.data
∂out/∂other = self.data

So by chain rule:

self.grad  += other.data * out.grad
other.grad += self.data  * out.grad

Implementation:

def __mul__(self, other):
    out = Value(self.data * other.data, (self, other), '*')

    def _backward():
        self.grad += other.data * out.grad
        other.grad += self.data * out.grad

    out._backward = _backward
    return out

Conceptually:

multiplication swaps the forward values of the operands,
then multiplies by the incoming gradient

Example:

out = a*b
a.grad receives b.data * out.grad
b.grad receives a.data * out.grad

45. Local backward for tanh

For:

out = tanh(self)

the derivative is:

d/dx tanh(x) = 1 - tanh(x)^2

Since:

out.data = tanh(self.data)

the local derivative is:

1 - out.data²

Implementation:

def tanh(self):
    x = self.data
    t = (math.exp(2*x) - 1) / (math.exp(2*x) + 1)
    out = Value(t, (self,), 'tanh')

    def _backward():
        self.grad += (1 - t**2) * out.grad

    out._backward = _backward
    return out

Equivalent form:

self.grad += (1 - out.data**2) * out.grad

Conceptually:

tanh scales the incoming gradient by the slope of tanh at the current input

Near zero, this slope is large.

Near saturation at -1 or +1, this slope is small.

46. Important correction: assign the function, do not call it

Karpathy hits a bug.

Incorrect:

out._backward = _backward()

This calls _backward immediately and stores its return value.

Since _backward returns None, this makes:

out._backward = None

Then later:

out._backward()

fails because None is not callable.

Correct:

out._backward = _backward

No parentheses.

Meaning:

store the function for later use

This distinction matters:

_backwards = function object
_backwards() = result of calling function

47. Manually calling _backward in reverse graph order

After adding local _backward functions, Karpathy rebuilds the neuron expression:

o = tanh(x1*w1 + x2*w2 + b)

All gradients start at zero.

To begin backprop:

o.grad = 1.0

Then he manually calls:

o._backward()

This backpropagates through the tanh node and sets:

n.grad = 0.5

Then:

n._backward()

This backpropagates through the plus node:

x1w1x2w2.grad = 0.5
b.grad = 0.5

Then:

x1w1x2w2._backward()

This backpropagates through another plus node:

x1w1.grad = 0.5
x2w2.grad = 0.5

Then:

x2w2._backward()
x1w1._backward()

This backpropagates through the multiplication nodes:

x2.grad = 0.5
w2.grad = 0.0

x1.grad = -1.5
w1.grad = 1.0

So the automated local functions reproduce the manual gradients.

But there is still a problem:

we are still manually deciding the order of backward calls

48. Need a correct global backward order

Backprop has an ordering requirement.

Before calling a node’s _backward, all later nodes that depend on it must already have contributed to its .grad.

In other words:

do not backpropagate through a node until its final gradient is complete

For a graph:

x → ... → y → ... → L

we need to run backward from:

L back toward x

So we need a way to order the graph.

This is where topological sorting enters.

49. Topological sort

A computation graph is a directed acyclic graph, or DAG.

A topological ordering arranges nodes so that:

children appear before the nodes that depend on them

For the forward graph:

inputs → intermediate nodes → output

a topological order lists inputs first and output last.

Then backprop can iterate over the list in reverse:

output → intermediate nodes → inputs

This gives a valid backward pass order.

50. Building the topological order

Karpathy implements a depth-first search:

topo = []
visited = set()

def build_topo(v):
    if v not in visited:
        visited.add(v)
        for child in v._prev:
            build_topo(child)
        topo.append(v)

Then:

build_topo(o)

This recursively visits every child before appending the current node.

The key invariant:

a node is appended only after all its children have been appended

So if o depends on n, and n depends on earlier nodes:

earlier nodes appear before n
n appears before o

Then:

reversed(topo)

gives the correct backprop order.

51. Manual global backward using topological order

The explicit procedure becomes:

o.grad = 1.0

for node in reversed(topo):
    node._backward()

This does the same thing as manually calling:

o._backward()
n._backward()
x1w1x2w2._backward()
x2w2._backward()
x1w1._backward()

but now the order is computed automatically from the graph.

This is the essence of an autograd engine:

build graph during forward pass
topologically sort graph
walk it backward
call local backward functions

52. Adding .backward() to Value

Now Karpathy hides the whole procedure inside the Value class.

The public method:

def backward(self):
    topo = []
    visited = set()

    def build_topo(v):
        if v not in visited:
            visited.add(v)
            for child in v._prev:
                build_topo(child)
            topo.append(v)

    build_topo(self)

    self.grad = 1.0

    for node in reversed(topo):
        node._backward()

Now users can simply call:

o.backward()

or later:

loss.backward()

and all gradients are populated automatically.

This matches the PyTorch-style API.

53. The current Value class conceptually

At this point, the class contains:

class Value:
    def __init__(self, data, _children=(), _op='', label=''):
        self.data = data
        self.grad = 0.0
        self._prev = set(_children)
        self._op = _op
        self.label = label
        self._backward = lambda: None

    def __add__(self, other):
        ...

    def __mul__(self, other):
        ...

    def tanh(self):
        ...

    def backward(self):
        ...

Each Value knows:

1. its numeric data
2. its gradient
3. its previous nodes
4. the operation that created it
5. how to backpropagate through that operation

This is a minimal scalar autograd engine.

54. A subtle bug: repeated use of the same node

Karpathy then shows a bug.

Example:

a = Value(3.0, label='a')
b = a + a
b.label = 'b'
b.backward()

Forward:

b = a + a = 6

Mathematically:

db/da = 2

because:

b = 2a

But the first implementation gives:

a.grad = 1

This is wrong.

55. Why the bug happens

In the expression:

b = a + a

both operands are the same object.

Inside addition’s backward function, if we wrote:

self.grad = 1.0 * out.grad
other.grad = 1.0 * out.grad

then both self and other refer to a.

So this does:

a.grad = 1
a.grad = 1

The second assignment overwrites the first.

But it should have accumulated both contributions:

a.grad = 1 + 1 = 2

Therefore the fix is:

self.grad += 1.0 * out.grad
other.grad += 1.0 * out.grad

not:

self.grad = 1.0 * out.grad
other.grad = 1.0 * out.grad

56. More general version of the same bug

Another example:

a = Value(-2.0)
b = Value(3.0)

d = a * b
e = a + b
f = d * e

f.backward()

Here a and b are each used in two branches:

a,b → d = a*b
a,b → e = a+b
d,e → f = d*e

During backprop:

f → e → a,b
f → d → a,b

Both branches contribute gradients to a and b.

If backward functions assign gradients instead of accumulating them, one branch overwrites the other.

Correct behavior requires summing contributions from all paths.

57. Multivariable chain rule: gradients accumulate

When a variable influences the output through multiple paths, the total derivative is the sum of all path contributions.

So:

gradient contributions add

Therefore every local _backward function should use:

+=

not:

=

Correct addition backward:

def _backward():
    self.grad += out.grad
    other.grad += out.grad

Correct multiplication backward:

def _backward():
    self.grad += other.data * out.grad
    other.grad += self.data * out.grad

Correct tanh backward:

def _backward():
    self.grad += (1 - t**2) * out.grad

This fixes repeated-node and multi-path graphs.

58. Why gradients start at zero

Gradient accumulation only works because gradients are initialized to zero:

self.grad = 0.0

Then every backward contribution adds on top:

initial grad = 0
first contribution added
second contribution added
third contribution added
...

So .grad becomes the total derivative.

61. Supporting constants in arithmetic

Current operations expect both operands to be Value objects.

This works:

a + b

But this does not yet work:

a + 1

because 1 is an integer and has no .data.

Fix:

other = other if isinstance(other, Value) else Value(other)

Inside __add__:

def __add__(self, other):
    other = other if isinstance(other, Value) else Value(other)
    out = Value(self.data + other.data, (self, other), '+')
    ...
    return out

Similarly inside __mul__:

def __mul__(self, other):
    other = other if isinstance(other, Value) else Value(other)
    out = Value(self.data * other.data, (self, other), '*')
    ...
    return out

Now these work:

a + 1
a * 2

because constants are automatically wrapped into Value.

62. Supporting reversed multiplication: 2 * a

After adding constants, this works:

a * 2

But this still fails:

2 * a

Reason:

a * 2 calls a.__mul__(2)
2 * a asks int.__mul__(a), and int does not know Value

Python offers a fallback method:

__rmul__

So implement:

def __rmul__(self, other):
    return self * other

Now:

2 * a

redirects to:

a * 2

and works.

63. Implementing exponentiation: .exp()

To build tanh from its formula, we need exponentials.

Add:

def exp(self):
    x = self.data
    out = Value(math.exp(x), (self,), 'exp')

    def _backward():
        self.grad += out.data * out.grad

    out._backward = _backward
    return out

Derivative:

d/dx eˣ = eˣ

Since:

out.data = eˣ

the local derivative is:

out.data

Then chain rule gives:

self.grad += out.data * out.grad

64. Implementing power: a ** k

Division can be expressed using powers.

Karpathy implements:

def __pow__(self, other):

where other must be a plain number:

assert isinstance(other, (int, float))

Forward:

out = Value(self.data ** other, (self,), f'**{other}')

Derivative rule:

d/dx xⁿ = n*xⁿ⁻¹

So backward:

def _backward():
    self.grad += other * (self.data ** (other - 1)) * out.grad

Then:

out._backward = _backward
return out

Now:

a ** 2
a ** -1

work.

65. Implementing division using power

Division:

a / b

can be rewritten:

a * (1/b)

and:

1/b = b⁻¹

So:

a / b = a * b⁻¹

Implementation:

def __truediv__(self, other):
    return self * other**-1

This reuses multiplication and power.

Because constants are wrapped as Value, division works with combinations like:

a / b
a / 2

Depending on reversed division support, 2 / a may require __rtruediv__, but the lecture’s immediate need is expressing the tanh formula.

66. Implementing negation and subtraction

Subtraction can be built from addition and negation.

Negation:

-a = -1 * a

Implementation:

def __neg__(self):
    return self * -1

Subtraction:

a - b = a + (-b)

Implementation:

def __sub__(self, other):
    return self + (-other)

This keeps the primitive operation set small.

Instead of writing a separate local backward rule for subtraction, reuse:

addition
multiplication by -1

68. Breaking tanh into smaller operations

Original neuron:

o = n.tanh()

New equivalent expression:

e = (2*n).exp()
o = (e - 1) / (e + 1)

This implements:

tanh(n) = (e^(2n) - 1) / (e^(2n) + 1)

Expected result:

same forward output
same gradients
larger computation graph

Forward output remains:

o ≈ 0.7071

The leaf gradients should still be:

x1.grad = -1.5
w1.grad = 1.0
x2.grad = 0.5
w2.grad = 0.0
b.grad  = 0.5

And they are.

69. What this proves

The same mathematical function can be represented at different graph granularities.

Version A:

n → tanh → o

Version B:

n → multiply by 2 → exp → subtract/add/divide → o

Both are valid.

The autograd engine only requires:

each operation knows its local derivative

Then chain rule handles the rest.

This is the central abstraction:

backprop does not care whether an operation is tiny or composite
as long as its local backward rule is correct

70. Practical takeaway about operation granularity

An autograd library designer chooses which operations are primitives.

For example, you could implement:

tanh as one primitive

or decompose it into:

exp, add, subtract, divide

Both can work.

Tradeoff:

larger primitive ops:
    fewer graph nodes
    need custom derivative formula

smaller primitive ops:
    more graph nodes
    derivatives may be simpler and reusable

Modern frameworks do both, depending on efficiency and numerical stability.

Micrograd is pedagogical, so clarity matters more than performance.
